# JaReader 发布前迭代计划：TTS 收敛与测试验证

更新时间：2026-07-24

## 目标

本轮目标是把 JaReader 推到可发布状态。范围包含三部分：

1. TTS 体验收敛：公开主线只保留系统 Google TTS，补齐声线选择指引和朗读文本清洗。
2. 发布前测试：验证高峰阶段的稳定性、内存开销、UI 流畅性，以及字典查询算法的速度和实用性。
3. 大模型罗马音注音：补齐句级罗马音注音能力，作为类似已有的大模型翻译功能的降低日语阅读门槛的辅助功能。

翻译交互已在上一轮完成三态重构。云 TTS 真实接入、逆向 TTS 不进入本轮发布前主线。

## 已确认素材

根目录已有同源测试小说：

- `败犬女主太多了 第9卷 日文版.epub`
- `败犬女主太多了_第9卷_日文版.txt`

这两份文件用于导入压力测试、EPUB/TXT 解析对照、随机定位查词测试和 TTS 文本清洗回归。

## 关键决策

### 1. 不做逆向免费网站 TTS

本轮不接入逆向免费网站 TTS。原因：

- 可用性、风控、验证码、接口变更风险不可控。
- 版权、服务条款和分发风险不适合发布前主线。
- 移动端一旦把逆向链路暴露给普通用户，后续维护成本很高。

Edge TTS、Azure TTS、Google Cloud TTS 继续作为隐藏或预留 provider 保留，但不在普通设置页展示，也不作为发布验收条件。

### 2. 公开主线只做系统 Google TTS

系统 Google TTS 是本轮唯一公开推荐方案。

设置页需要明确说明：

- 推荐安装或启用 Google 文字转语音。
- 需要下载日语语音数据。
- 不同日语声线效果差异明显。
- 默认声线在低语速下可能出现颤音，建议用户在系统 TTS 设置中尝试男性声线或其他日语声线。
- JaReader 调用系统 TTS，实际可选声线取决于手机系统和已安装语音包。

如果应用内可以稳定读取 `expo-speech.getAvailableVoicesAsync()` 的日语语音列表，可以增加应用内声线选择；如果不同 Android 设备表现不稳定，则先做设置页指引，不强行承诺应用内枚举完整声线。

### 3. TTS 重复读优先按 ruby/注音残留处理

“汉字词后面紧跟假名导致同一个词读两遍”优先假设为 EPUB ruby 或 TXT 注音标记被抽取进正文，例如：

- `漢字（かんじ）`
- `漢字《かんじ》`
- `｜漢字《かんじ》`
- EPUB `<ruby><rb>漢字</rb><rt>かんじ</rt></ruby>` 被解析为 `漢字かんじ`

不能简单删除“汉字后接假名”，因为普通送假名是日语正文的一部分，例如 `食べる`、`思った`、`読んだ`。

本轮策略：

- 优先在 EPUB 解析阶段正确处理 `<ruby>`，正文显示保留或降级为基底文本，TTS 输入只读基底文本。
- 对 TXT 和已扁平化文本增加 TTS-only 正规化函数，先处理带括号或青空文库风格 ruby 标记。
- 对无括号的 `漢字かな` 连写只做检测和日志采样，不自动删除，除非后续精准度测试证明有高置信规则。

### 4. 罗马音注音进入本轮

罗马音注音不再作为“暂缓”，而是进入本轮发布前任务，但第一版只追求稳定可用，不追求完美逐字符 ruby 排版。

功能目标：

- 在阅读页增加注音开关或按钮。
- 对当前句生成罗马音注音。
- 结果缓存到本地，避免重复请求或重复计算。
- 以稳定布局展示在原文上方或句卡内部。
- 用户可随时隐藏注音。

实现建议：

- 优先复用现有 LLM 配置或翻译 API Key，不新增复杂账号体系。
- LLM 输出约束为 JSON，避免自由文本难解析。
- 缓存 key 建议使用 `bookId + sentenceIndex + sourceTextHash`。
- 如果 LLM 不可用，功能应同翻译一起降级为“未配置大模型服务”，不影响阅读、查词、TTS 。
- 展示层第一版可做 token/短语级注音，不要求精确到每个字符。

建议 JSON 格式：

```json
{
  "items": [
    { "text": "彼女", "romaji": "kanojo" },
    { "text": "は", "romaji": "wa" }
  ]
}
```

验收：

- 未配置 API Key 时不发请求，并显示可理解提示或禁用入口。
- 当前句点击注音后能显示罗马音。
- 翻页后注音状态不会错挂到新句。
- 回到已注音句子时优先读取缓存。
- LLM 返回非 JSON、漏字段、文本对不齐时不崩溃。
- 长句注音不会遮挡正文、译文或查词卡片。

### Agent B：TTS 文本清洗与 ruby 残留处理

状态：待开始

写入范围：

- `src/services/tts.ts`
- 新增 `src/services/tts-normalizer.ts`
- `src/services/parser/epub.ts`
- `src/services/parser/txt.ts`
- 必要时新增小型纯函数测试脚本

任务：

- 新增 `normalizeTextForSpeech(text)`，只影响 TTS 输入，不改变阅读正文和查词文本。
- 清理青空文库风格 ruby：`｜漢字《かな》` 转为 `漢字`。
- 清理括号注音：高置信场景下 `漢字（かな）`、`漢字《かな》` 转为 `漢字`。
- EPUB ruby 解析优先提取 `rb`，避免 `rt` 被并入朗读文本。
- 对疑似无括号扁平 ruby 做采样记录，不自动删除。

验收：

- `食べる`、`思った`、`読んだ` 等正常送假名不被破坏。
- 常见 ruby 标记不会在 TTS 中重复朗读。
- 正文显示、查词、翻译缓存 key 不因 TTS 清洗改变。

### Agent C：稳定性与性能测试工具

状态：待开始

写入范围：

- `scripts/`
- `docs/testing-checklist.md`
- 可选新增 `docs/performance-test-report.md`

任务：

- 设计并实现导入压力测试脚本或开发者测试入口。
- 覆盖首次打开词典导入、TXT 导入、EPUB 导入、句子窗口加载、翻译缓存批量读取。
- 记录耗时、处理条数、峰值内存估算、失败点和重复运行波动。
- 提供 Android Studio / adb 手动采样流程。

建议指标：

- 字典初始化总耗时。
- 小说导入总耗时、分句数量、SQLite 写入耗时。
- `loadSentenceWindow` p50 / p95 耗时。
- 翻译缓存 `getCachedTranslations` p50 / p95 耗时。
- Android 真机 `adb shell dumpsys meminfo com.jareader.app` 导入前、导入中、导入后采样。
- UI 主线程是否出现明显卡顿，必要时用 Android Studio Profiler 辅助确认。

验收：

- 测试流程可以由 agent 或维护者重复执行。
- 报告能指出是否存在明显内存臃肿点。
- 如果发现 UI 卡顿，需要定位到词典导入、小说解析、SQLite 写入或窗口读取中的具体阶段。

### Agent D：字典查询精准度与速度基准

状态：待开始

写入范围：

- `scripts/`
- `src/services/dictionary.ts`
- `src/services/tokenizer.ts`
- 可选新增 `docs/dictionary-benchmark-report.md`

任务：

- 从测试小说中按固定随机种子抽取句子和字符位置。
- 过滤标点、空白和明显非日语位置。
- 调用当前查词核心函数，记录命中词、候选片段、查询耗时。
- 输出 JSON/CSV，便于人工标注“实用 / 不实用 / 未收录 / 误命中”。
- 对比至少两种策略：token 原始命中、当前字符覆盖最长匹配。

重点问题：

- 当前速度变慢是否来自查询精度提升。
- 慢点是 SQLite 多次同步查询、候选片段过多、prefix 搜索，还是 fallback `LIKE '%query%'`。
- 是否可以通过 LRU 缓存、候选批量查询、减少 prefix 次数、限制候选长度来恢复速度。

初步判断：

- 查词变慢不一定是“降低内存开销”的必然代价。
- 更可能是字符覆盖最长匹配引入了更多 SQLite 查询。
- 优先优化查询次数和缓存，而不是引入大型内存 trie。
- 大型内存索引只有在基准证明 SQLite 查询无法满足交互速度时再考虑。

验收：

- 输出至少 500 次随机点击样本的耗时分布。
- p95 查询耗时目标先定为 50ms 以内；如果真机较弱，可放宽但需要说明。
- 人工抽样至少 100 条，给出实用率和主要误命中类型。
- 给出是否需要继续优化算法的明确结论。

## 分阶段测试要求

每个 Agent 完成后必须提交局部验证结果，主 agent 不接受“只改不测”的交付。

### 每个模块完成后的最低检查

- 运行 `git status --short`，说明自己改了哪些文件。
- 运行 `npx tsc --noEmit`，如失败必须说明是否由本模块导致。
- 对新增纯函数运行脚本级验证或提供最小样例。
- 说明 mock、预留、等待真实 API Key 或等待真机验证的部分。
- 说明需要维护者在 Android 真机上验证的具体操作。
- 说明是否和其他模块存在写入文件或状态语义冲突。

### TTS 模块额外检查

- 验证未配置云 Key 时不会发起云请求。
- 验证 `normalizeTextForSpeech` 不改变阅读正文。
- 用至少 10 条 ruby/注音样例验证 TTS 清洗。
- 用至少 10 条正常送假名样例验证不误删。

### 罗马音注音模块额外检查

- 验证 JSON parser 能处理合法输出、空数组、漏字段、非 JSON。
- 验证缓存命中不会重复请求 LLM。
- 验证翻页后 loading 结果不会覆盖当前句。
- 验证长句布局在竖屏和横屏下不遮挡主要阅读内容。

### 查词基准模块额外检查

- 固定随机种子，保证结果可复现。
- 输出样本总数、命中率、p50/p95/p99 查询耗时。
- 人工标注样本保留原句、点击位置、候选词、最终词条。
- 若提出优化，必须先给优化前基线。

### 稳定性测试模块额外检查

- 记录测试设备或模拟环境。
- 记录导入前、导入中、导入后的内存采样。
- 记录首次词典初始化耗时和是否阻塞 UI。
- 记录 TXT / EPUB 导入耗时和分句数量。
- 记录窗口加载和缓存读取耗时。

## 推荐执行顺序

1. Agent C 先建立稳定性和性能测试框架。
2. Agent D 建立查词精准度与速度基准，确认当前变慢原因。
3. Agent B 做 TTS 文本清洗，避免重复朗读。
4. Agent A 收敛设置页和公开 TTS 文案。
5. Agent E 实现罗马音注音第一版。
6. 主 agent 汇总报告，更新发布测试清单。
7. 维护者 Android Studio 真机总测。

这个顺序的原因是：先有测量工具，再改优化点，能避免凭感觉做性能判断；TTS 文本清洗和设置页文案风险较低，可在性能基线明确后并行或顺序收尾。罗马音注音涉及 reader 页面和缓存表，建议放在基础测试框架之后做，便于立刻验证新增状态机。

## 发布前验收标准

### 自动检查

```powershell
npx tsc --noEmit
```

如涉及 Android 原生构建：

```powershell
cd android
.\gradlew.bat assembleDebug
```

### 整体交付前检查

主 agent 集成所有模块后必须完成：

- 汇总每个 Agent 的修改文件、实现行为、预留项和局部测试结果。
- 运行 `npx tsc --noEmit`。
- 运行 `git diff --check`。
- 如时间允许运行 `cd android; .\gradlew.bat assembleDebug`，超时或失败必须记录原因。
- 更新 `docs/testing-checklist.md`。
- 更新或新增性能/精准度报告文档。
- 明确列出还需要维护者真机验证的事项。

### 真机检查

- 首次启动和词典初始化不会明显卡死。
- 导入测试 TXT 成功，进度和分句正常。
- 导入测试 EPUB 成功，正文和插图不异常。
- 随机翻页、目录跳转、书签仍正常。
- 点按查词响应速度可接受，命中结果对学习者有实用价值。
- 滑动选词不明显误触发纵向滚动。
- 系统 Google TTS 朗读正常。
- 设置页能指导用户选择更合适的日语声线。
- ruby 或注音残留不会导致 TTS 明显重复朗读。
- 罗马音注音可显示、可隐藏、可缓存，异常输出不影响阅读。

## 暂不进入本轮

- 逆向免费网站 TTS。
- Azure / Google Cloud TTS 真实 Key 验证。
- 大型内存词典索引或 trie，除非基准测试证明确有必要。
