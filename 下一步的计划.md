模块一：TTS 引擎重构（高优先级，P0）
现状：依赖 python edge-tts 服务端，项目无力自建服务；kokoro 内置尝试失败（闪退）。
目标：构建 “云端高质量优先 + 本地离线兜底” 的双引擎模式。

探索 Edge TTS 网页端 API（尝试性）：调研 Edge TTS 网页版（speech.platform.bing.com）的 WebSocket/HTTP 请求格式。若能在 Android 端直接模拟请求且不违反机器人协议，网站本身运营稳定，则将其封装为第一优先级引擎（无需自建服务器）。若存在跨域或签名限制，则跳过此步。

集成 Google TTS 离线引擎（兜底方案）：将目前已有的指引用户安装Google TTS 引擎封装为稳定的降级方案。
若有其它可考虑的内置方案也可尝试。

代码清理：移除项目中所有对本地 Python 服务端的硬编码依赖和启动检测逻辑。

模块二：大模型翻译交互优化（高优先级，P0）
现状：翻译按钮是“切换式（开/关）”，翻页不关闭，切换句子需点两次，体验割裂。
目标：改为 “智能替换式” 交互逻辑。

状态机重构：定义翻译卡片的三种状态（隐藏、显示当前句、显示历史句）。

具体逻辑：

当前卡片隐藏时：点击按钮 → 翻译当前句段 → 卡片弹出显示，按钮弹起状态为“开启”。

翻页时：卡片保持显示，但内容暂不更新（显示上一页句子）。

当前卡片显示且句子相同时：点击按钮 → 关闭卡片（隐藏）。

当前卡片显示且切换到新句段时：点击按钮 → 立刻检查本地 LRU 缓存（键值为句子Hash）→ 若有缓存直接显示，若无则发起新的大模型请求并显示新内容。

缓存机制（目前已有，但要保护防破坏）：翻译结果持久化到本地数据库（Room 或 DataStore），避免重复请求浪费 Token。

模块三：文本点选查询精度与滑动体验修缮（高优先级，P0）
现状：点击分词不准，模糊查询效果差；滑动选择似乎缺少 Y 轴判定，手动选不中。
目标（仅建议）：放弃“先分词后查询”的传统路线，转向 “点击点中心扩散 + 最长匹配/多待选均保留” 算法，并修复触摸事件。

交互逻辑调整（点击查询）：

获取用户点击在文本中的字符索引（Index）。

查询算法（替代旧版模糊查询）：以该索引为中心，向右扩展（优先），同时向左扩展，动态查找内置词典中存在的最长连续字符串（例如：点击「問い合わせ」，优先命中「問い合わせ」整体，而非「問い」）。

如果最长匹配长度仅为 1（单字），则显示该字的释义（这符合用户预期，因为日语汉字查字率很高）。


交互逻辑修复（滑动选择）：

重写相关逻辑，严格加入垂直方向（Y轴）的偏移量判定。

设定相关阈值，防止点击和滑动判断错误。

模块四：大模型罗马音注音功能（中优先级，P1）
现状：tts效果不好的话用户听不清，想通过标注罗马音辅助认读。
目标：提供可选的“罗马音注音”开关，基于 LLM 生成。

UI 布局：在翻译按钮旁增加独立的 “注音（振り仮名）”开关按钮。

Token 节省策略（关键决策）：

作为单独的功能按钮放在翻译按钮旁边，或是和翻译的功能合并，在内置提示词里的同时要求大模型按某种格式要求给出罗马音。取决于哪种更节省token。

渲染逻辑：解析返回的罗马音，以悬浮注音（Ruby Text） 的形式显示在原文句段每一行的上方（类似小学生拼音），注音必须与原文文字严格对齐。

模块五：TTS 朗读前的智能断句预处理（中优先级，P1）
现状：长句直接丢给 TTS，缺少停顿，听感机械且难以消化。
目标：在调用 TTS 引擎之前，增加句法分割层。

实现 JapaneseSplitter 工具类：基于日语标点（。、、）和接续助词（て、に、を、が）进行分块。

逻辑：对于超过 15 个字符（或特定音拍）的句子，拆分为 2-3 个短句，按顺序送入 TTS 队列播放，中间插入 200ms 的静音间隔。
