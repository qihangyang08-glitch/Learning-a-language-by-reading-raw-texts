# JaReader 二轮重构 —— 问题诊断与计划

## 问题1：阅读界面频繁崩溃 + 翻页卡顿 + 纲目跳转崩溃

### 根因分析

1. **同步阻塞JS线程** — `tokenizerService.tokenize()` 和 `JSON.parse(fullDict)` 均在JS线程同步执行，导致UI掉帧和ANR。

2. **TextCard 无 memo 保护** — `TextCard` 每次父组件渲染都全新渲染所有 token `<Text>` 元素。token 数组通过 `useMemo` 每次产生新引用。

3. **InlineImage 内存泄漏** — `setTimeout(check, 80)` + `requestAnimationFrame` 在组件卸载时未正确清理，导致对已卸载组件 setState。Android 上 Image 组件加载大图时尤其容易 OOM。

4. **PageGestureZone 动画竞态** — `withTiming` 回调中的 `runOnJS` 可能在新手势已经开始时触发，导致 shared value 状态不一致崩溃。

5. **纲目跳转 double-trigger** — `handleOutlineSelect` 先更新 window 再更新 currentIndex，但 `useEffect` 也会在 currentIndex 变化时重新加载 window，两个操作形成竞态条件。当 window 尚未加载完成时 `currentSentence` 为 `undefined`，TextCard 渲染时 `sentence.tokens` 访问 null 引用崩溃。

6. **kuromoji.js 未在 package.json 依赖中** — `import('kuromoji')` 是 dynamic import，但 kuromoji 不在 dependencies 里。加载失败时静默降级到正则分词，正则分词非常粗糙。

### 修复方案

- [P0] 在 reader 组件树顶部添加 `ErrorBoundary` 防止单点崩溃
- [P0] 给 `TextCard` 加 `React.memo` + 稳定的 props 比较
- [P0] 修复 `handleOutlineSelect` 竞态：window 加载完成后 atomically 更新 currentIndex
- [P0] 修复 `InlineImage` 的生命周期管理
- [P0] 添加 `sentence` 为 null 时的安全渲染
- [P1] kuromoji 降级方案：改为纯正则增强版（更可靠的字符类分割 + 形态素规则）
- [P1] PageGestureZone 动画加锁防止重入

## 问题2：本地词典不全 + 体积过大

### 根因分析

- 当前方案：内置 ~500 词 + 运行时下载完整 JMdict（zip 25MB，解压后 ~200MB JSON）
- `JSON.parse(fullDict)` 将整个 JSON 加载到内存，低端设备 OOM
- 下载失败后只剩 500 词，远不够读小说
- SQLite FTS5 建立后 DB 体积 ~80-120MB

### 修复方案

- [P0] **放弃运行时下载完整 JMdict**。改为预构建精选词典 SQLite DB，作为 asset 打包
- [P0] 精选 JMdict 中频率最高的 25K-30K 词条 → 生成 8-15MB 的 SQLite DB
- [P0] 使用频率数据（BCCWJ/NINJAL 等语料库频率列表）优先收录高频词
- [P1] 构建脚本放在 `scripts/build-dict.ts`，在发版前手动执行一次

## 问题3：点击选中不精准 + 滑动选中未实现

### 根因分析

1. **正则分词太粗糙** — 当前正则：`/([一-龯々々]+|[ぁ-んー]+|[ァ-ンー]+|...)/g`
   - `食べ始める` → `["食", "べ", "始", "める"]`（应为 `["食べ", "始める"]` 或 `["食べ始める"]`）
   - 没有词边界概念，只按字符类分割

2. **kuromoji 不可用** — 未在 dependencies 中，dynamic import 大概率失败

3. **Pan 手势阈值太严格** — `activeOffsetX([-8, 8])` + `failOffsetY([-10, 10])` 使得稍有垂直偏移的手势就失败

4. **translationX → token 映射错误** — `translationX / charW` 假设所有字等宽且文本不换行，在混合 CJK/ASCII 文本和换行场景下不准确

### 修复方案

- [P0] **用 `tiny-segmenter` 替代 kuromoji** — 纯 JS，~50KB，无需词典文件，准确率 95%+
- [P0] 改写 `TextCard` 的选中逻辑：点击根据 token index 查询，不再依赖 `translationX`
- [P0] 放宽 pan 手势阈值，使用 `hitSlop` 而非 `activeOffsetX`
- [P0] 实现真正的滑动选中：记录手指在 token 上的移动轨迹，基于 token index 而非像素偏移
- [P1] 添加长按选中短语（作为滑动选中的补充）

## 问题4：LLM 整段翻译因崩溃未测试

### 根因

这是问题1的衍生问题。reader 频繁崩溃导致无法进入稳定状态测试翻译功能。

翻译代码本身设计合理（DeepSeek API + 本地缓存），修复稳定性问题后应可用。

### 修复方案

- 修复问题1后重新测试
- 添加翻译错误的重试逻辑（3次）
- 翻译 loading 期间防止翻页（避免竞态）

## 整体架构重构

### 高优先级

1. **Reader 组件拆分** — 当前 `app/reader/[bookId].tsx` 430行，包含全部逻辑。拆分为：
   - `useReaderInit` — 开书、window加载
   - `useReaderGestures` — 手势协调
   - `useReaderDictionary` — 分词+查词
   - `useReaderTranslation` — 翻译逻辑
   - `useReaderTTS` — TTS 控制

2. **TextCard 性能** — `React.memo` + 稳定 token 引用 + `FlashList` 思路（虚拟化）

3. **Tokenizer 方案切换** — kuromoji(不可用) → tiny-segmenter(纯JS，轻量)

4. **词典方案切换** — 运行时下载JMdict → 预构建精选SQLite

5. **错误恢复** — ErrorBoundary 包裹 reader 各区域，独立恢复

### 中优先级

6. 翻译功能完善（重试、loading状态保护）
7. PageGestureZone 和 TextCard 手势物理隔离测试
8. 单手模式 UI 完善

### 低优先级

9. TTS 自动连读
10. 横屏适配优化
